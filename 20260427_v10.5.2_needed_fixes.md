## ❌ Blocking Issues Found

### 1. Phase 2 (MintFromConversion) - Hash Validation Failure
**Symptom**: `lock receipt hash mismatch (tampering detected)`

**Root Cause**: The `ConversionLock` struct does not store `TxID` from Phase 1, but the hash verification in Phase 2 requires it to match.

**Current State**:

**Phase 2 Receipt** (requires TxID but lock doesn't store it):

## 🔧 Required Fixes for v10.5.3

### Option A: Store TxID in ConversionLock (Recommended)
**Pros**: 
- Maintains hash-based tamper detection
- No breaking changes to hash algorithm
- Full audit trail with original transaction ID

**Implementation**:
1. Add `TxID string` field to `ConversionLock` struct in `chaincode/assets/conversion_lock.go`
2. Store `ctx.GetStub().GetTxID()` in Phase 1 (LockGOForConversion)
3. Include `TxID` in `ConversionLockReceipt` struct
4. Phase 2 (MintFromConversion) retrieves TxID from stored lock and uses in receipt

**Code changes**:
```go
// chaincode/assets/conversion_lock.go
type ConversionLock struct {
    // ... existing fields ...
    TxID             string  `json:"TxID"`              // ADD THIS
    LockReceiptHash  string  `json:"LockReceiptHash"`
    // ... rest of fields ...
}

// chaincode/contracts/conversion_crosschannel.go - Phase 1
lock := assets.ConversionLock{
    // ... existing fields ...
    TxID:            ctx.GetStub().GetTxID(),  // ADD THIS
    LockReceiptHash: fmt.Sprintf("%x", lockReceiptHash[:]),
}
```

### Option B: Modify Hash to Exclude TxID
**Pros**:
- Simpler - no struct changes
- Still provides tamper detection

**Cons**:
- Loses transaction traceability
- Breaking change to hash algorithm

**Implementation**:
```go
// Remove ctx.GetStub().GetTxID() from hash calculation in both phases
receiptData := fmt.Sprintf("%s:%s:%s:%s:%s:%s",
    lockID,
    goAssetID,
    sourceChannel,
    destChannel,
    ownerMSP,
    sourceIssuerMSP)
    // Removed: ctx.GetStub().GetTxID()
```

### Option C: Store Full Receipt in Lock
**Pros**:
- Most robust - entire receipt stored and retrieved
- No manual hash reconstruction needed

**Cons**:
- Larger data storage
- More complex struct changes

## 💡 Lessons Learned

1. **Pointer types**: Fabric contract API doesn't support pointer types in return values - must use non-pointer primitives
2. **omitempty**: Doesn't work with non-pointer zero values - schema validation will fail; remove tag entirely
3. **UTF-8 BOM**: JSON files must not have BOM - causes Fabric parse errors
4. **Transient data**: Required for all privacy-sensitive write operations, not Args
5. **Hash integrity**: Must store ALL hash inputs or exclude non-reproducible fields (like unique TxID per call)
6. **MSP mapping**: Cross-channel conversions require careful source/destination MSP mapping via DestinationOwnerMSP field
